# rc003 note id prefix resolution 需求澄清

日期：2026-04-26

## 需求背景

用户希望 Zembra 像 git、docker 一样，允许通过 UUID / ID 的前几位定位对应笔记，减少日常命令里输入完整 32 位 note id 的负担。

## 已确认需求

| 项目 | 结论 |
| --- | --- |
| 功能目标 | 用户输入 note id 前缀时，系统可以解析到唯一匹配的笔记。 |
| 最短前缀 | 最短 4 位。少于 4 位的输入不参与短 ID 匹配。 |
| 匹配对象 | 只匹配未删除笔记。软删除笔记不参与所有常规操作的短 ID 解析。 |
| 完整 ID | 完整 32 位 note id 继续支持，行为保持兼容。 |
| 字符规则 | 接受十六进制字符前缀，大小写输入统一按小写处理。 |
| 无匹配行为 | 返回明确的未找到错误。 |
| 多匹配行为 | 返回明确的冲突错误，并提示用户输入更长前缀。 |
| 冲突提示 | 推荐展示候选笔记的短 id 和内容摘要，帮助用户继续定位。 |

## 范围边界

- 本需求聚焦 note id 前缀解析能力。
- 默认不改变 note id 生成规则，仍使用当前 UUID hex 字符串。
- 默认不让软删除笔记参与匹配。
- 不在本需求中新增恢复软删除笔记的能力。
- 具体 CLI 命令如何接入该解析能力，进入设计文档阶段再确认。

## 验收标准

- 输入长度不少于 4 位且唯一匹配未删除笔记时，可以解析到完整 note id。
- 输入完整 note id 时，原有精确查询行为不被破坏。
- 输入少于 4 位的短前缀时，系统返回清晰错误。
- 输入无匹配前缀时，系统返回清晰错误。
- 输入匹配多条未删除笔记的前缀时，系统返回清晰冲突错误。
- 已软删除笔记不会被短前缀解析命中。
